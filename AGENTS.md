## Epics, issues, and agent teams

### Phases map to epics

Before starting any phase, file a beads epic for it and one issue per numbered step:

```bash
bd epic create --title "Phase N: <name>" --description "<goal of this phase>"
bd create --title "Step N: <summary>" --epic <epic-id> --type=task --priority=2
# one issue per numbered step in the phase
```

Do not begin a phase without all its issues filed. Do not begin a step without claiming its issue (`bd update <id> --claim`).

### Agent teams for parallel steps

Use `TeamCreate` to spawn parallel agent workstreams for steps within a phase that are independent. Each agent:
- Claims its own beads issue before starting
- Reads only the files and spec sections relevant to its task (grep for what you need; do not load the full codebase)
- Closes its issue when done; escalates blockers immediately rather than retrying indefinitely
- Does NOT touch files outside its assigned scope

Do not spawn agents for steps that share mutable state (same file, same module).

If an agent makes 3 attempts at the same error without progress, stop and escalate to the user rather than retrying.

## Testing requirements

**Test-forward.** Write tests before or alongside the implementation. An issue is not closeable until its tests pass. Test writing is never deferred.

**Independent oracle.** Every test must verify against an oracle that does not use the code under test.
- Serializing then deserializing with the same function is not a test — it proves only internal consistency.
- For serde/wire-type tests: hand-write the expected JSON directly from the spec text, commit it as a fixture in `crates/jmap-chat/tests/fixtures/`, and assert round-trip equality against that file.
- For HTTP client tests: use committed `.json` response fixtures or a local mock HTTP server (`wiremock`). Never use live network calls in tests.
- If a fixture was generated by a script (Python `json.dumps`, `jq`, etc.), commit the script alongside the fixture.

**Never weaken tests.** Do not skip, `#[ignore]`, delete, or soften assertions to make tests pass. Fix the code. If the fix is out of scope, file a new beads issue and escalate.

**No fabricated results.** Never report tests as passing unless you have run them and seen zero failures.

## Subagent guidance

Spawn subagents aggressively for research, exploration, and independent workstreams.
Do not spawn subagents for tasks that share mutable state (e.g., editing the same file).
Each subagent reads only what it needs — grep for specific symbols rather than loading full files.
